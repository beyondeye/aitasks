## Exploration Summary

- **Focus:** Add Codex CLI's `default_mode_request_user_input` feature flag to the project-level Codex config generated by `ait setup`.
- **Context:** Running `codex features enable default_mode_request_user_input` updates the user config at `~/.codex/config.toml`. For aitasks projects, setup currently creates or merges `.codex/config.toml` from `seed/codex_config.seed.toml`, staged into `aitasks/metadata/codex_config.seed.toml` by `install.sh`.
- **Key findings:**
  - `seed/codex_config.seed.toml` is the source template for project Codex settings.
  - `install.sh::install_seed_codex_config` copies that seed into `aitasks/metadata/` during framework installation/staging.
  - `.aitask-scripts/aitask_setup.sh::setup_codex_cli` creates `.codex/config.toml` from the staged seed or merges missing settings via `merge_codex_settings`.
  - Existing setup integration coverage lives in `tests/test_agent_instructions.sh`, especially `create_codex_staging` and the `setup_codex_cli` tests.

## Desired Change

Update the Codex setup pipeline so `ait setup` automatically adds the Codex CLI under-development feature flag:

```toml
[features]
default_mode_request_user_input = true
```

This should apply to new project installs and to existing `.codex/config.toml` files when `ait setup` is rerun, without overwriting unrelated user settings.

## Implementation Notes

- Add the feature flag to `seed/codex_config.seed.toml`.
- Ensure the staged metadata seed path receives the same content through the existing install copy path.
- Update setup tests so fresh config creation and merge behavior assert the feature flag is present.
- Verify idempotency: rerunning setup should not duplicate or disturb existing TOML content.

## Non-Goals

- Do not remove the current requirement for aitasks Codex skill workflows to run through plan-mode helpers yet.
- Do not change skill rendering or workflow dispatch in this task.
- Treat removal of the plan-mode requirement as a follow-up only after verifying this Codex feature flag actually makes `request_user_input` available in default mode.
