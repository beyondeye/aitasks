## Symptom

`ait ide` exits immediately after showing the tmux status bar for a fraction of a
second. The user sees the tmux top bar with the window list flash briefly and then
the shell prompt returns. `./ait ide` (invoked directly, bypassing the global
shim) works correctly. `tmux new-session -s aitasks -n monitor 'ait monitor'`
(direct invocation) also works. Only the shim-mediated `ait ide` fails.

Discovered after laptop restart; was working before. The restart clears
`/tmp/tmux-*` sockets and forces a fresh tmux server, which exposed the bug —
presumably the user previously had a shell session where `_AIT_SHIM_ACTIVE` was
unset for some other reason.

## Root Cause

The global shim generated by `aitask_setup.sh` (heredoc at lines 560-647) has a
recursion guard that leaks to child processes:

```bash
# .aitask-scripts/aitask_setup.sh, lines 565-578
if [[ "${_AIT_SHIM_ACTIVE:-}" == "1" ]]; then
    echo "Error: ait dispatcher not found in any parent directory." >&2
    exit 1
fi
export _AIT_SHIM_ACTIVE=1          # exported to all children

dir="$PWD"
while [[ "$dir" != "/" ]]; do
    if [[ -x "$dir/ait" && -d "$dir/.aitask-scripts" ]]; then
        exec "$dir/ait" "$@"       # BUG: guard not unset before exec
    fi
    dir="$(dirname "$dir")"
done
```

The `unset _AIT_SHIM_ACTIVE` call is missing before the walk-up `exec` on line 575.
The sibling "ait setup bootstrap" path at line 639 already does this correctly
(`unset _AIT_SHIM_ACTIVE` then `exec "$PWD/ait" setup`) — only the primary
walk-up path was missed.

## Failure Chain for `ait ide`

1. User runs `ait ide` → shim at `~/.local/bin/ait` executes
2. Shim sets `_AIT_SHIM_ACTIVE=1` (exported), execs project-local `./ait ide`
3. Project `ait` dispatcher runs `.aitask-scripts/aitask_ide.sh`
4. `aitask_ide.sh:100` runs `exec tmux new-session -s aitasks -n monitor 'ait monitor'`
5. Tmux server inherits env including `_AIT_SHIM_ACTIVE=1` and passes it to the
   new pane shell
6. Pane runs `bash -c 'ait monitor'` → `ait` resolves via PATH to the shim
7. Shim sees `_AIT_SHIM_ACTIVE=1` → prints
   `"Error: ait dispatcher not found in any parent directory."` → exits 1
8. Only window in the session exits → session destroyed
9. Tmux client attached briefly, session is gone, client detaches
   (user's `~/.config/tmux/tmux.conf` has `detach-on-destroy off` which falls back
   to detach when no other sessions exist)
10. User sees the status bar flash, then shell prompt returns

## Reproduction

Confirmed in this environment:

```bash
$ _AIT_SHIM_ACTIVE=1 /home/ddt/.local/bin/ait monitor
Error: ait dispatcher not found in any parent directory.
exit=1
```

## Fix

### 1. Fix the shim template in `.aitask-scripts/aitask_setup.sh`

Add `unset _AIT_SHIM_ACTIVE` before the walk-up `exec` so the guard does not
leak into child processes spawned by the project-local `ait`:

```bash
while [[ "$dir" != "/" ]]; do
    if [[ -x "$dir/ait" && -d "$dir/.aitask-scripts" ]]; then
        unset _AIT_SHIM_ACTIVE   # <-- add this line
        exec "$dir/ait" "$@"
    fi
    dir="$(dirname "$dir")"
done
```

Rationale: the recursion guard is only meaningful for the in-process walk-up and
bootstrap flow. Once the shim has found a project-local `ait` and is about to
hand off via `exec`, the guard is no longer relevant and must not pollute the
child's environment (including any subprocess, tmux session, or shell the child
launches).

### 2. Add a regression test in `tests/test_global_shim.sh`

The existing test file unsets `_AIT_SHIM_ACTIVE` at line 9 ("Clear shim guard
that may leak from parent environment"), which is why this bug was not caught.

Add a new test case that:
- Sets up a temp project dir with `.aitask-scripts/` and a trivial `ait` script
  inside it that prints its argv and an env snapshot (e.g. `env | grep -c
  _AIT_SHIM_ACTIVE`)
- Invokes the shim from inside that temp project
- Asserts that the project-local `ait` received no `_AIT_SHIM_ACTIVE` in its
  environment (the count should be `0`)

This covers the "guard leaks through exec into child `ait` calls" scenario that
was previously untested.

### 3. Re-installation notes

Users currently affected need to regenerate their shim by running `ait setup`
(or the `install_global_shim` function directly) after this fix lands. Consider
whether the changelog entry for this fix should explicitly call out that
existing users will need to re-run setup to pick up the fixed shim.
