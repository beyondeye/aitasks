we created significant infrastructure with aiexplains generated data that allow to drill down for each file which tasks generated them and see tha task plans of them. the motivation was to help the user, using the codebrowser tui + the aitask-explain skill, to understand the source of the existing code and WHY it exists, in addition on how it works. Now I realized that this information can be actually very useful also for the code agent itself, when working on a specific file or when trying to understand the broader design decisions behind a code file design, and how a specific file fit in a specific feature architecture. For example, if i want to write a task to document a new skill or a new batch command, i can generate much better docs if I have the full context of the why a command exists and how it integrates in the full feature that spans multiple components. example of this are the feature implemented by task 355. or the feature implemented by task 303. this are complex features that spans multiple files and component. I would like for a way to integrate the two systems, What a user currently can do manually is go to the codebrowser. manually search in the source code lines for plans  numbers most relevant and feed their numbers in the task definition, hoping that the aitask-pick skill will be able to extract them (not garanteed becauses aiexplains data has all the plans pre-extracted and organized, while the raw data can be already tar zipped in old.tar.gz files. so this will probably not work. also there is the issue of context creep. the information contained in thse plans will contain many details that are necessairly important for the task at hand and this can fill the llm context and confuse the code agent. this is more of a task for a dedicated subagent that will extract the actual relevant information from the plans and return what is actually needed, but how can i trigger a custom subagent from a skill, and do it in a way that is compatible to multple codeagents (claudecode, opencode, codex, geminicli?) I am not sure that this is possible, perhaps hthe more practical solution would be some "preprocess" skill that will update the task with criical information that is needed from the referenced plan that define the architecture. by the way this information SHOULD BE VERIFIED because this is information from HISTORICAL task plans: it is true that the aiexplains data resolve only the most recent plans that are connected to existing codeline, but still there can be for sure obsolete information referenced by plans. AND the final problem is that is a flow that involve the user manually selecting the task plan number to reference in the task definition. this should be automated in some way. we can beuild a script that extract for each file the list of referenced plan (the same list that is available in the codebrowser, with plans ordered by importance according of number of contributed lines. but how the new preprocess skill will now which file, of the files need to be changed by the aitask should be have this treatment? THERE SEVERAL OPEN QUESTIONS HERE. please brainstorm several design options and once the design is finalized create multiple child tasks for implementation.
