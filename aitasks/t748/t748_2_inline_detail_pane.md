## Context

Part of t748 (parent: `aitasks/t748_browsable_node_graph.md`, plan: `aiplans/p748_browsable_node_graph.md`). This child adds an inline detail pane to the Graph tab, mirroring the Dashboard tab's 40/60 horizontal split, so the focused DAG node's metadata + "Generated by" operation block + dimensions render reactively to the right of the DAG.

The Dashboard tab already does this for `NodeRow` (`_show_node_detail` at `brainstorm_app.py:3710-3788`). After t749 (now archived), `_show_node_detail` also renders an operation block (operation badge with `OP_BADGE_STYLES`, Agents row, When row, `"Press 'o' for operation details"` hint). The Graph-tab inline pane must show the same content.

To avoid drift between the two panes, this child also extracts a shared widget-list helper from `_show_node_detail` rather than cloning the body.

## Implementation Plan

1. **Refactor `_show_node_detail` to extract `_render_node_detail_widgets(node_id) -> tuple[str, list[Widget]]`** ‚Äî pure helper returning `(title_text, widget_list)` for the metadata block + "Generated by" operation block + Dimensions block. Reads `self.session_path`. The helper does NOT mount widgets ‚Äî it returns them ready to mount.
2. **Make `_show_node_detail` a thin caller:** clear `#dash_node_info`, set `#dash_node_title`, mount the helper's widgets.
3. **Add `_show_dag_node_detail(node_id)`** ‚Äî analogous thin caller for `#dag_node_title` and `#dag_node_info`.
4. **Restructure `tab_dag` compose** (currently `brainstorm_app.py:2514-2515`):
   ```python
   with TabPane("(G)raph", id="tab_dag"):
       with Horizontal(id="dag_split"):
           yield DAGDisplay(id="dag_content")
           yield VerticalScroll(
               Label("", id="dag_node_title"),
               Container(id="dag_node_info"),
               id="dag_detail_pane",
           )
   ```
5. **Add CSS** for `#dag_split`, `#dag_content`, `#dag_detail_pane` mirroring `#dashboard_split` widths (60/40 split for the Graph tab ‚Äî DAG ASCII art needs more horizontal space than NodeRow list).
6. **Add `DAGDisplay.FocusChanged(node_id)` message class** (in `brainstorm_dag_display.py`) and post it from `action_next_node` / `action_prev_node` / `action_open_node`. Also from the four arrow-key actions added by t748_1 (touch those if already merged; if not yet merged, drop a note in Final Implementation Notes so t748_1's actions also post `FocusChanged` when implemented).
7. **Initial population:** post `FocusChanged` from `DAGDisplay.load_dag()` after the first focus is set so `_show_dag_node_detail` runs immediately on tab entry.
8. **Wire `@on(DAGDisplay.FocusChanged)` handler** in `BrainstormApp` that calls `self._show_dag_node_detail(event.node_id)`.
9. **Track focused-graph-node id for dimension drill-in.** Cleanest approach: rename `_current_dashboard_node_id` ‚Üí `_current_focused_node_id` and set it from both `_show_node_detail` and `_show_dag_node_detail` so `on_dimension_row_activated` (the handler that opens `SectionViewerScreen` for a `DimensionRow`) works uniformly. Alternative: add `_current_graph_node_id` sibling and have the handler check both.
10. **Confirm `o` from the Graph tab still pushes `OperationDetailScreen`.** No change needed ‚Äî `DAGDisplay.OperationOpened` ‚Üí `@on` handler at `brainstorm_app.py:3863-3879` is independent of the compose change. Just verify by manual test.

## Verification Steps

1. Launch `ait brainstorm` on a session with several nodes.
2. Tab to (G)raph ‚Äî DAG renders on the left (~60%), detail pane on the right (~40%).
3. Focus a node (arrows or `j`/`k`) ‚Äî the right pane updates immediately with: Description / Parents / Created / Generated-by block / Dimensions.
4. The Generated-by block matches the Dashboard's content for the same node: operation badge colored by `OP_BADGE_STYLES`, Agents row, When row, `"Press 'o' for operation details"` hint.
5. Press `o` ‚Äî `OperationDetailScreen` opens for the focused node's group.
6. Focus a `DimensionRow` in the right pane and press Enter ‚Äî filtered `SectionViewerScreen` opens for the Graph-focused node (this confirms step 9 above works).
7. Confirm the Dashboard tab still works (the refactor must not regress it).
