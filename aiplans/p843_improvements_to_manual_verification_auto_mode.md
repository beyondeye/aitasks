# t843 — Auto-execution mode for manual_verification tasks

## Context

When the user ran t787 (a `manual_verification` task) with codex 5.5 instead of
Claude Code, codex spontaneously synthesised an *executable* plan: it drove the
brainstorm TUI inside a tmux session, sent keystrokes, fabricated test data,
and produced pass/fail verdicts without a per-item AskUserQuestion loop. The
user found this valuable and wants it as a **first-class option** for
`manual_verification` tasks across all code agents — including Claude Code,
where the current Manual Verification Procedure
(`.claude/skills/task-workflow/manual-verification.md`) always goes straight
into the interactive Pass / Fail / Skip / Defer loop.

The new mode must be **optional** (not the default), **per-item granular**
(some checklist items remain genuinely human-only), and **profile-aware** so
power users can opt in/out without per-task prompting.

User decisions (locked in via planning AskUserQuestion + follow-up
clarifications):

1. **Trigger point** — both up-front (before the per-item loop) AND as a
   per-item escape hatch inside the loop.
2. **Two auto-verification strategies** are supported:
   - **Impromptu (default)** — agent executes checklist items inline,
     picking the verification approach on the fly per item. The plan file
     is written at the **end** as a retroactive record of what was actually
     run (this is what codex 5.5 spontaneously did for t787).
   - **Pre-built plan** — agent designs a per-item execution plan up front,
     optionally requests user approval via EnterPlanMode / ExitPlanMode,
     then executes. The plan file captures both the prospective plan and
     the execution log.
3. **Plan artifact** — both strategies persist to the same path:
   `aiplans/p<id>_manual_verification_auto.md` (or
   `aiplans/p<parent>/p<id>_manual_verification_auto.md` for child tasks).
   Structurally: impromptu plans contain only an execution log; pre-built
   plans contain both the upfront plan and the log.
4. **Fallback** — best-effort per item; items the agent cannot automate are
   left `pending` (impromptu) or marked `defer` with a reason in the
   pre-built plan, then fall through to the interactive loop.
5. **Default behaviour** — `ask` (prompt fires on every manual_verification
   task) for both `default` and `fast` profiles. **The "yes" default in the
   prompt is impromptu** — pre-built requires an explicit second click.
6. **Per-item escape hatch** — surface as an extra verb `auto` in the
   existing "Other" free-text branch. The per-item `auto` verb is **always
   impromptu** (no plan design or approval gate — it's a single item).

## Scope

This is a **procedure / workflow change**: no new bash helpers and no Python
changes. The existing `aitask_verification_parse.sh set <file> <idx> <state>
--note "<text>"` already supports a `--note` on every state (verified in
`.aitask-scripts/aitask_verification_parse.py:243`), so auto-verification can
record *how* an item was verified directly on the checklist line without new
infrastructure.

Per CLAUDE.md "Working on Skills" — Claude Code is the source of truth. Codex

## Detailed design

### New profile key

Single key, five values:

| Value | Up-front prompt | Strategy when auto runs | Approval gate |
|-------|-----------------|-------------------------|---------------|
| `ask` (default — key absent or explicit) | Yes (3-way choice: impromptu / pre-built+approve / skip) | Whatever user picks | Only if user picks "pre-built" |
| `never` | No | n/a — straight to interactive | n/a |
| `impromptu` | No | Impromptu (always) | None |
| `prebuilt_approve` | No | Pre-built (always) | Required |
| `prebuilt_autorun` | No | Pre-built (always) | Skipped |

In `profiles.md` Schema Reference, add the row:

```
| `manual_verification_auto_mode` | string | no | `"ask"` (default), `"never"`, `"impromptu"`, `"prebuilt_approve"`, `"prebuilt_autorun"` — controls the Step 1.5 up-front auto-execution prompt and selects the strategy when auto runs | Manual Verification Step 1.5 |
```

Note that the per-item `auto` verb in Step 2 is **always** available

### New procedure file — `auto-verification.md`

Handles both strategies. Skeleton:

```markdown
# Auto-Verification Procedure

Automated verification for `manual_verification` tasks. Invoked from
`manual-verification.md` Step 1.5 (whole checklist) and Step 2 (single item,
via the `auto` verb). Supports two strategies:

- **Impromptu** — execute each item inline, picking the verification
  approach on the fly. The plan file is written **at the end** as a
  retroactive record of what was actually run. This is the default and
  matches the codex 5.5 behaviour observed on t787.
- **Pre-built** — design a per-item execution plan up front, write it to
  the plan file, optionally request approval via EnterPlanMode /
  ExitPlanMode, then execute and append an Execution Log.

## Input context

| Variable | Type | Description |
|----------|------|-------------|
| `task_file` | string | path to the manual_verification task file |
| `task_id` | string | task identifier (e.g. `787` or `571_7`) |
| `is_child` | boolean | whether `task_id` is a child task |
| `parent_id` | string/null | parent number if `is_child`, else null |
| `strategy` | string | `"impromptu"` or `"prebuilt"` |
| `approval_required` | boolean | only meaningful when `strategy == "prebuilt"`; ignored otherwise. true → EnterPlanMode + ExitPlanMode |
| `single_item` | int/null | when set, restrict to that one item index (Step 2 per-item path); null = whole checklist. **Forces `strategy = "impromptu"`** regardless of caller's value. |
| `active_profile` | object | loaded execution profile |

## Plan file path

- parent task: `aiplans/p<task_id>_manual_verification_auto.md`
- child task: `aiplans/p<parent_id>/p<task_id>_manual_verification_auto.md`

Metadata header mirrors the parent / child header templates from
`planning.md`'s **Save Plan to External File** section.

## Procedure

### 1. Enumerate target items

Run `aitask_verification_parse.sh parse <task_file>`. If `single_item` is
set, filter to that index. Otherwise keep every item currently in state
`pending` or `defer`.

### 2. Strategy branch

#### 2a. Impromptu (`strategy == "impromptu"`)

For each target item, in order:

1. **Pick a verification approach on the fly.** Heuristics:
   - File inspection — text/YAML/JSON the item references → `cat` / `grep`
     / `head` / a Python one-liner.
   - CLI invocation — the item names a script, subcommand, or `ait` verb →
     invoke via Bash and check exit code / output.
   - TUI interaction — TUIs like `ait brainstorm` / `ait monitor` → spawn
     a detached tmux session, drive it with `tmux send-keys`, capture pane
     content with `tmux capture-pane -p`. Reuse the session-creation
     pattern at `.aitask-scripts/lib/tmux_bootstrap.sh:158`;
     `.aitask-scripts/monitor/monitor_app.py:84,1388` shows the Textual-key
     → tmux-key mapping.
   - Test data fabrication — create scratch files under
     `${TMPDIR:-/tmp}/auto_verify_<task_id>_<idx>/` (cleaned at end). Never
     mutate user-owned files in `aitasks/` / `aiplans/` other than the
     checklist itself.
   - Not automatable — UX judgement, visual rendering, multi-screen flows
     → mark `defer` with a reason. No execution attempted.

2. **Execute** and capture stdout/stderr/exit-code.

3. **Mark state** (same per-state helpers as the interactive loop):
   - Pass criterion met → `aitask_verification_parse.sh set <task_file>
     <idx> pass --note "auto: <one-line summary>"`.
   - Pass criterion violated → `aitask_verification_followup.sh --from
     <task_id> --item <idx>` (existing Fail flow; parse
     `FOLLOWUP_CREATED` / `ORIGIN_AMBIGUOUS` / `ERROR` exactly as in
     `manual-verification.md` step 2 Fail branch).
   - Cannot decide / blocked → `aitask_verification_parse.sh set
     <task_file> <idx> defer --note "auto: <why blocked>"`.

4. **Append to in-memory log** (kept until step 3 below).

#### 2b. Pre-built (`strategy == "prebuilt"`)

1. Read all target items.
2. Design a per-item plan using the same heuristics as 2a step 1, but
   write nothing yet.
3. Write the plan file (Write tool) with sections:
   - `## Pre-built Auto-Execution Plan` — numbered list, one entry per
     target item: `<idx>. [<expected_state>] <item text>` with indented
     bullets `- Strategy: …`, `- Action: …`, `- Pass criterion: …`,
     `- Fail / defer fallback: …`.
   - `## Cleanup` — scratch files / tmux sessions to remove afterwards.
4. **Approval gate.** If `approval_required == true`:
   - EnterPlanMode.
   - ExitPlanMode for user approval (the plan file content is what the
     user reviews).
   - If user rejects → delete the just-written plan file, return to the
     caller without mutating any item state.
5. Execute each item per the approved plan, marking state via the same
   helpers as 2a step 3. After each item, append a `### Item <idx>` block
   to a `## Execution Log` section of the plan file (Edit tool, growing
   the file in place).

### 3. Write the plan file (impromptu path) or finalise (pre-built path)

- **Impromptu:** Write the plan file (Write tool) now, with sections:
  - `## Execution Log` — for each item processed: `### Item <idx>` with
    `- Item text: …`, `- Approach: …`, `- Action run: …` (bash / tmux),
    `- Output (trimmed): …`, `- Verdict: pass | fail | defer`.
  - `## Cleanup` — scratch files / tmux sessions removed (or to remove if
    the next step is interrupted).
- **Pre-built:** plan file already exists from step 2b; ensure the
  Execution Log appended in step 2b step 5 is complete.

### 4. Cleanup

Remove scratch dirs and any tmux sessions created during execution.

### 5. Commit plan file

```bash
./ait git add aiplans/<plan_path>
./ait git commit -m "ait: Add manual-verification auto-execution plan for t<task_id>"
```

### 6. Return

Return to `manual-verification.md`:
- Step 1.5 caller → fall through to step 2; items still `pending` /
  `defer` are handled interactively.
- Step 2 single-item caller (the `auto` verb path) → return to step 2's
  loop top so the checklist re-renders with the new state.
```
