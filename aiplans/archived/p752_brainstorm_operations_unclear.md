# Plan: Context-aware "Operation Help" modal in brainstorm Actions wizard

## Context

Task: `t752_brainstorm_operations_unclear.md` (bug, medium/medium).

The task description names "ait monitor" but the relevant tab is in the
`ait brainstorm` TUI (Actions tab → Step 1 — Select Operation). The user
clarified that they want a **context-aware keyboard shortcut** that, when an
operation row is focused in Step 1, opens a modal with a detailed
description and use cases for that specific operation — including which
operation operates on the proposal vs the plan, and which operations bring
over the parent node's proposal and/or plan.

The operations and their actual I/O were verified by reading the agent
prompt templates in `.aitask-scripts/brainstorm/templates/`:

| Op | Reads from base/parent node(s) | Produces |
|---|---|---|
| explore | YAML metadata + proposal + plan (if exists) + reference files | new node: proposal (no plan) |
| compare | YAML metadata only across N selected nodes | comparison matrix (no new node, no edits) |
| hybridize | YAML metadata + proposal from N source nodes | new node: merged proposal |
| detail | YAML metadata + proposal of selected node | new plan attached to that node |
| patch | YAML metadata + plan (edit target) + proposal (read-only, impact analysis) | patched plan + impact verdict |

## Approach

Single-file change in `.aitask-scripts/brainstorm/brainstorm_app.py`:

### 1. Add an `_OPERATION_HELP` content constant

A module-level dict keyed by `op_key`, each entry shaped:

```python
{
    "title": "Explore — Architecture Explorer",
    "summary": "<one-paragraph plain-prose description>",
    "reads_from_parent": ["…", "…"],
    "produces": ["…", "…"],
    "use_cases": ["…", "…"],
}
```

Keys covered: the 5 design ops (`explore`, `compare`, `hybridize`, `detail`,
`patch`) and the 5 session-lifecycle ops (`pause`, `resume`, `finalize`,
`archive`, `delete`). Lifecycle entries can omit `reads_from_parent` /
`produces` (no I/O over node content) and rely on `summary` + `use_cases`.

The agent-template prompts in `.aitask-scripts/brainstorm/templates/*.md`
remain the source of truth for full agent behavior; this dict is a
condensed user-facing summary surfaced in the wizard.

**Source-trace comments are required.** Each design-op entry MUST be
preceded by a code comment naming the source template file and the
specific section(s) the I/O contract was derived from, so a future
contributor opening the source file can verify and re-derive the help text
without consulting archived plans. Style:

```python
_OPERATION_HELP = {
    # Source: .aitask-scripts/brainstorm/templates/explorer.md
    # I/O contract derived from "## Input" (reads parent YAML metadata,
    # proposal, plan-if-exists, reference files) and "## Output" (produces
    # new node YAML metadata + proposal markdown; no plan).
    "explore": { ... },

    # Source: .aitask-scripts/brainstorm/templates/comparator.md
    # I/O from "## Input" — explicitly metadata-only ("do not read
    # proposals, plans, or codebase files") — and "## Output" (comparison
    # matrix + delta summary; no new node).
    "compare": { ... },

    # Source: .aitask-scripts/brainstorm/templates/synthesizer.md
    # I/O from "## Input" (reads source nodes' YAML + proposals) and
    # "## Output" (new node with merged proposal; parents = [all sources]).
    "hybridize": { ... },

    # Source: .aitask-scripts/brainstorm/templates/detailer.md
    # I/O from "## Input" (reads node YAML + proposal + reference files)
    # and "## Output" (single plan markdown; proposal & metadata unchanged).
    "detail": { ... },

    # Source: .aitask-scripts/brainstorm/templates/patcher.md
    # I/O from "## Input" — plan is the edit target, proposal is read-only
    # for impact analysis — and "## Output" (patched plan + impact verdict
    # NO_IMPACT/IMPACT_FLAG; optional metadata update on impact).
    "patch": { ... },
}
```

Plus a top-of-block comment naming the templates directory:

```python
# Help text condensed from the agent prompt templates in
# .aitask-scripts/brainstorm/templates/*.md (one prompt per design op).
# When the prompts change, update the per-entry source comments below
# AND the corresponding "summary"/"reads_from_parent"/"produces" fields.
_OPERATION_HELP = { ... }
```

Session-lifecycle entries (`pause`/`resume`/`finalize`/`archive`/`delete`)
have no template file backing them — their behavior lives in the runtime
session machinery. Each gets a short comment naming the runtime source
instead, e.g.:

```python
    # Source: BrainstormApp._is_session_op_disabled + session lifecycle
    # transitions in brainstorm_session.py (status machine: init/active/
    # paused/completed/archived).
    "pause": { ... },
```

Pointing at the canonical state machine site keeps the same trace
discipline even when no template file applies.

### 2. New `OperationHelpModal(ModalScreen)` near the other modals

Pattern modeled on `LogModal` / `NodeDetailScreen` already in the file.

```python
class OperationHelpModal(ModalScreen):
    BINDINGS = [
        Binding("escape", "close", "Close", show=False),
        Binding("?", "close", "Close", show=False),
    ]

    def __init__(self, op_key: str):
        super().__init__()
        self.op_key = op_key

    def compose(self) -> ComposeResult:
        with Container(id="op_help_dialog"):
            info = _OPERATION_HELP.get(self.op_key)
            yield Label(
                f"Operation: {info.get('title', self.op_key) if info else self.op_key}",
                id="op_help_title",
            )
            yield VerticalScroll(
                Markdown(self._render_markdown(info), id="op_help_content"),
                id="op_help_scroll",
            )
            yield Label("[dim]Esc / ? close[/]", id="op_help_footer")

    def _render_markdown(self, info: dict | None) -> str:
        if not info:
            return f"*No help available for `{self.op_key}`.*"
        parts = [info["summary"], ""]
        if info.get("reads_from_parent"):
            parts.append("## Reads from base/parent node(s)")
            parts.extend(f"- {x}" for x in info["reads_from_parent"])
            parts.append("")
        if info.get("produces"):
            parts.append("## Produces")
            parts.extend(f"- {x}" for x in info["produces"])
            parts.append("")
        if info.get("use_cases"):
            parts.append("## Use cases")
            parts.extend(f"- {x}" for x in info["use_cases"])
        return "\n".join(parts)

    def action_close(self) -> None:
        self.dismiss(None)
```

CSS: piggyback on the existing modal-dialog look (`#op_help_dialog`
mirrors `#log_modal_dialog` rules near line 1588). Add a small CSS block
near the existing modal CSS.

### 4. Scope visibility in `check_action`

Extend `check_action` (line 1662) to gate the `?` binding to Step 1 of the
Actions tab:

```python
if action == "op_help":
    try:
        tabbed = self.query_one(TabbedContent)
    except Exception:
        return None

        return None
```

Result: `?` appears in the footer only on Actions Step 1, hidden everywhere

we also need the wizard-step check; an explicit branch in `check_action` is
the simplest fit.)

### 5. Action handler `action_op_help`

```python
def action_op_help(self) -> None:
    from textual.actions import SkipAction
    try:
        tabbed = self.query_one(TabbedContent)
    except Exception:
        raise SkipAction
    if tabbed.active != "tab_actions" or self._wizard_step != 1:
        raise SkipAction
    focused = self.focused
    if not isinstance(focused, OperationRow):
        raise SkipAction
    if focused.op_key not in _OPERATION_HELP:
        raise SkipAction
    self.push_screen(OperationHelpModal(focused.op_key))
```

`raise SkipAction` is consistent with the priority-binding guidance in
CLAUDE.md (so other handlers / default key behavior still get a chance if
the guard misses).

     not the literal `?` character.

  separate appendix or external doc. This matches the
  `feedback_source_comments_for_derived_help_text` memory and ensures the
