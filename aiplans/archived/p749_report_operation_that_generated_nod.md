# Plan: Report operation that generated each node in `ait brainstorm`

## Context

In the brainstorm TUI today, every node carries a `created_by_group` field
(e.g. `explore_001`, `patch_002`) that names the operation group which
produced it, but **the user never sees that information surfaced anywhere
useful** тАФ only a one-liner `Group: <name>` in the dashboard right pane.
There is no way from a node to see:

- which **operation** ran (explore / compare / hybridize / detail / patch)
- the **inputs** (mandate, merge rules, patch request, target sections,
  source nodes, dimensions) that drove that operation
- the **agents** involved, their **statuses**, **logs**, and **outputs**
- when it was run and what HEAD it was based on

The data exists on disk but is scattered across `<agent>_input.md`,
`<agent>_output.md`, `<agent>_status.yaml`, `<agent>_log.txt`. There is also
a latent gap: `br_groups.yaml` is initialized empty by `init_session` but
**never populated** when an operation is launched. Every read site (Status
tab, Recent Operations) currently sees an empty dict.

This task adds operation provenance to the brainstorm TUI in three places:
a per-node badge in the DAG view, a richer "Generated by" block in the
dashboard right pane, and a new dedicated **Operation Detail** screen
opened with `o` on a focused node. As a foundation, it also makes
`br_groups.yaml` actually get written when operations run.

## Design Decisions (confirmed with user)

1. Surfaces: **DAG node-box badge** + **Dashboard right-pane** (NOT Node
   Detail Modal, NOT Compare tab).
2. Detail depth: **Full dedicated screen** with Overview tab + per-agent
   tabs (each agent tab has Input / Output / Log sub-sections).
3. **No data duplication** in `br_groups.yaml`. Use a typed reference
   format that **pinpoints existing on-disk data** (agent input/output/log
   files, node files) rather than copying values. `br_groups.yaml` stores
   only the spec-defined fields (`operation`, `agents`, `status`,
   `created_at`, `head_at_creation`, `nodes_created`).
4. Key binding: **`o`** on a focused node opens the Operation Detail
   screen. Footer must show the new key AND any pre-existing
   `show=False` keys in the same tab/screen we touch (per recent feedback
   memory: "TUI footer must surface keys (existing + new)").
5. DAG box layout: **add a new badge row** (5 rows total, was 4).
6. Bootstrap node: pressing `o` on `n000_init` shows a minimal Operation
   Detail with the initializer agent (if a proposal-file import was used);
   for blank-init it shows the `initial_spec`.

## Reference format (the pinpoint primitive)

Define a small, typed reference struct in a new module
`brainstorm_op_refs.py`:

```python
@dataclass(frozen=True)
class OpDataRef:
    """Pointer to a piece of data on disk within a brainstorm session.

    Resolves relative to the crew worktree (session_path).

    kind:    "agent_input" | "agent_output" | "agent_log"
             | "node_proposal" | "node_plan" | "node_metadata"
             | "session_spec"
    target:  agent name (e.g. "explorer_001a") OR node id (e.g. "n003_alpha")
             OR "" for session-level refs.
    section: optional Markdown "## Header" string to scroll-to inside the
             referenced file. None means show the whole file.
    """
    kind: str
    target: str
    section: str | None = None

def resolve_ref(session_path: Path, ref: OpDataRef) -> str: ...
def file_for_ref(session_path: Path, ref: OpDataRef) -> Path: ...
```

Plus operation-aware helpers (no group-state required, derived from
operation type + agent list):

```python
def list_op_inputs(group_info: dict) -> list[OpDataRef]: ...
def list_op_outputs(group_info: dict) -> list[OpDataRef]: ...
def list_op_logs(group_info: dict) -> list[OpDataRef]: ...
def list_op_definition(group_info: dict) -> list[OpDataRef]:
    """The operation's "definition" refs тАФ head_at_creation node and
    nodes_created. Returned as node_metadata refs."""
```

Section anchors per op (so the Overview tab can show the user's mandate
without rendering all the boilerplate of the input.md):

| operation | section to highlight |
|-----------|-----------------------|
| explore   | `## Exploration Mandate` |
| compare   | `## Comparison Request` |
| hybridize | `## Merge Rules`        |
| detail    | (whole input тАФ no key user param) |
| patch     | `## Patch Request`      |
| bootstrap | `## Mandate` (initializer) |

This is the "reference format" the user asked for: data lives once on
disk; the screen displays it via these refs; nothing is duplicated.
