# t794 — Brainstorm OperationDetailScreen loading indicator

## Context

Pressing `o` on a node in the brainstorm TUI (either from the dashboard
node list or from the DAG view) pushes `OperationDetailScreen` to show
operation details. The modal's `compose()` synchronously reads
`br_groups.yaml`, then iterates per-agent and reads each agent's input,
output, and log files via `resolve_ref(...)` / `read_log_tail(...)`.
With several agents and non-trivial logs, that can take a few seconds,
during which the TUI shows nothing — the user can't tell whether `o`
was registered.

Goal: provide immediate visual feedback (<200ms) after `o` so the user
knows the keypress was registered and the modal is preparing.

## Approach

Use the task's "approach 1" (Textual `notify(...)` before
`push_screen(OperationDetailScreen(...))`). Minimal change, zero risk
to existing modal functionality. The notification appears within one
event-loop tick, well under 200ms, and auto-dismisses after `timeout`
seconds — it does not need to be tied to the modal's lifecycle.

The same notification is fired at both call sites (DAG view + node
row) so the behavior is identical regardless of which view triggered
the keypress.

We deliberately do NOT pursue approach 2 (mount a `Static("Loading…")`
inside the modal and populate widgets in `on_mount` /
`call_after_refresh`). That requires refactoring `compose()` into a
two-phase mount and changes the modal's internal lifecycle — out of
scope for this small-effort task and surfaced as a possible
follow-up in the task notes.

## Changes

File: `.aitask-scripts/brainstorm/brainstorm_app.py`

Two call sites, both currently doing only `self.push_screen(
OperationDetailScreen(...))`.

1. **`on_dag_display_operation_opened`** (line ~4200)

   Before `push_screen(...)` add:
   ```python
   self.notify("Opening operation details…", timeout=2)
   ```

2. **`on_node_row_operation_opened`** (line ~4296)

   Same single-line addition before `push_screen(...)`.

Both sites end up like:

```python
@on(DAGDisplay.OperationOpened)
def on_dag_display_operation_opened(
    self, event: DAGDisplay.OperationOpened
) -> None:
    """Open OperationDetailScreen from DAG view ('o' key)."""
    self.notify("Opening operation details…", timeout=2)
    self.push_screen(
        OperationDetailScreen(event.group_name, self.session_path)
    )
```

(The notification fires from `BrainstormApp` (App-level), which is
where `self.notify` is already used widely in this file — see e.g.
lines 1334, 2875.)

## Verification

Manual smoke (the task requires manual TUI verification):

1. Launch a brainstorm session that has at least one operation group
   with several agents and reasonably large logs (so the modal open
   is visibly slow without the indicator).
2. From the **node list** dashboard, focus a node generated by such
   an operation and press `o`. Confirm: the "Opening operation
   details…" toast appears immediately, then the modal renders.
3. From the **DAG view** (Graph tab), focus a node and press `o`.
   Confirm the same toast → modal sequence.
4. Confirm the modal still works the same as before (tabs, agent
   panes, close via Esc/`q`/Close button).
5. Confirm no regressions in surrounding `o`-key paths
   (`OperationHelpModal`, `SectionViewerScreen` for `p`/`l`).
