# t1181 — Fix stale agent-attribution plan-mode exemplar

## Context

`t1171` stopped the framework from injecting `/plan` mode when spawning Codex
agents. That made one sentence in the Agent Attribution procedure factually
wrong — it cites **Codex CLI** as the exemplar of an agent that "runs initial
workflow steps in plan mode".

Current text at `.claude/skills/task-workflow/agent-attribution.md:5`:

> **When to execute:** At the start of implementation, after plan mode has been
> exited. This timing is critical because some code agents (e.g., Codex CLI) run
> initial workflow steps in plan mode, which is read-only and cannot write
> metadata.

The **general claim is still true** — Claude Code's planning phase genuinely is
a read-only plan mode, and the deferred-write timing this procedure describes is
still required. So this is an **exemplar fix**, not a removal: the sentence and
its timing rationale stay.

This was deliberately carved out of t1171 during risk-mitigation design so that
change stayed scoped to the Codex launch path.

## Approach

Edit the **canonical source only**, then rerender. Per CLAUDE.md, `.claude/skills/`
is the source of truth; the rendered copies under `.claude/`, `.agents/`, and
`.opencode/` are never hand-edited.

### New wording

> **When to execute:** At the start of implementation, after plan mode has been
> exited. This timing is critical because on code agents with a plan mode (e.g.,
> Claude Code), the workflow's planning phase runs read-only and cannot write
> metadata.

Rationale for this phrasing:
- Keeps the generic claim ("code agents with a plan mode") as the load-bearing
  statement, with a single concrete anchor — consistent with the
  one-or-two-anchor variant of the genericization rule in
  `aidocs/framework/adding_a_new_codeagent.md` §23b.
- Names Claude Code, which is currently and verifiably correct.
- Preserves the read-only / cannot-write-metadata rationale verbatim in substance.

read-only plan mode — is directly observable in this very session.
