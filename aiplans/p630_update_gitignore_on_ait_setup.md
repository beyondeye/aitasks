# t630: Add `__pycache__/` to `.gitignore` during `ait setup`

## Context

On a fresh `ait setup`, Python TUI tools under `.aitask-scripts/` (board, codebrowser, monitor, minimonitor, settings, brainstorm, etc.) generate `__pycache__/` directories at runtime. None of these are currently added to the project's `.gitignore`, so after the user runs any TUI the directories show up as untracked in `git status` — visual noise, and a footgun for `git add -A`.

`commit_framework_files()` already filters pycache artifacts out of its own commit list (regex `(^|/)__pycache__/|\.py[co]$|\.pyd$`, lines 2423–2425), but that filter is defensive — it doesn't prevent the files from being listed as untracked. The correct fix is a `.gitignore` rule.

**Where it fits:** `setup_draft_directory()` already demonstrates the pattern — it appends `aitasks/new/` to `.gitignore` during setup (lines 1213–1246). We add a sibling helper for `__pycache__/`. The user asked for the rule to be "committed together with other framework files … after user approval" — so the new helper **does not commit on its own**. Instead, it only modifies `.gitignore`, which is already in `commit_framework_files`'s `check_paths`. That function then picks up the modification via `git ls-files --modified` and folds it into the single approval-gated framework commit.

## Files to modify

- `.aitask-scripts/aitask_setup.sh` — add `setup_python_cache_gitignore()` and wire it into `main`.

No changes to `install.sh`: it runs before any Python has executed (no `__pycache__` exists yet), and its `commit_installed_files` doesn't modify `.gitignore`. Adding the rule during `ait setup` is the right moment, per the user's ask.

No `seed/` template needed: the `.gitignore` is built incrementally by setup helpers, not copied from a template.

## Implementation

### Step 1 — Add the helper

In `.aitask-scripts/aitask_setup.sh`, insert a new function right after `setup_draft_directory` (ends at line 1246) and before the `check_latest_version` comment at line 1248:

```bash
# --- Python cache artifact gitignore rule ---
# Runs after setup_draft_directory (which ensures .gitignore exists).
# Does NOT commit: the modification is picked up by commit_framework_files()
# and included in its approval-gated commit.
setup_python_cache_gitignore() {
    local project_dir="$SCRIPT_DIR/.."
    local gitignore="$project_dir/.gitignore"

    # Already present?
    if [[ -f "$gitignore" ]] && grep -qxF "__pycache__/" "$gitignore"; then
        success "Python cache rule already in .gitignore"
        return
    fi

    info "Adding __pycache__/ to .gitignore (Python cache artifacts)..."

    if [[ -f "$gitignore" ]]; then
        echo "" >> "$gitignore"
        echo "# Python cache artifacts (generated by aitasks Python TUIs)" >> "$gitignore"
        echo "__pycache__/" >> "$gitignore"
    else
        {
            echo "# Python cache artifacts (generated by aitasks Python TUIs)"
            echo "__pycache__/"
        } > "$gitignore"
    fi

    success "Python cache rule added to .gitignore"
}
```

Design notes (mirrors `setup_draft_directory` for consistency):
- Uses `grep -qxF` for exact-line match (portable, macOS-safe — no PCRE).
- Create-or-append pattern: defensive against the rare case where `.gitignore` was removed between `setup_draft_directory` and this call.
- **No `git add`/`git commit`** — this is the key departure from `setup_draft_directory`. The modification is left unstaged; `commit_framework_files` will detect it as modified (since `.gitignore` is in `check_paths`, line 2390) and include it in the approval-gated framework commit.

### Step 2 — Wire it into `main`

In `main()` (around line 2820), insert the call immediately after `setup_draft_directory`:

```bash
    setup_draft_directory
    echo ""

    setup_python_cache_gitignore
    echo ""

    setup_id_counter
    echo ""
    # … rest unchanged
```

This placement ensures:
- `.gitignore` already exists (created by `setup_draft_directory` if needed).
- The modification is uncommitted when `commit_framework_files` runs later (line 2859), so it's picked up as a "modified" framework file and gets the user-approval prompt.
