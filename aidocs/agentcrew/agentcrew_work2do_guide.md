# AgentCrew work2do Authoring Guide

This guide explains how to write `work2do` files for AgentCrew agents. A work2do file is the task specification that an agent executes — it describes *what* to do and includes *checkpoints* where the agent calls lifecycle operations to report status, check for commands, and stay coordinated with the runner.

## Key Concept: Abstract Operations and Instructions Mapping

Work2do files use **abstract operation names** (like `report_alive`, `check_commands`) rather than concrete script calls. This keeps work2do templates reusable across different crew configurations.

The mapping from abstract names to concrete CLI commands is provided in each agent's `_instructions.md` file, which is auto-generated by `ait crew addwork`. The agent reads its instructions file to know exactly which commands to call.

**Why this separation?** Different crews may use different agent types, different CLI paths, or different wrapper scripts. By keeping the work2do abstract, the same template can be used across multiple crew configurations — only `_instructions.md` needs to change.

## Lifecycle Operations

These are the abstract operations an agent should call at checkpoints. The correct CLI syntax for each is listed below.

### `report_alive` — Heartbeat and Progress Message

Signal that the agent is still active. Include a human-readable message describing current activity.

```bash
ait crew status --crew <crew_id> --agent <agent_name> heartbeat --message "Processing step 3 of 10"
```

**When:** At every checkpoint. This is the minimum required operation — without heartbeats, the runner will mark the agent as stuck after the timeout (default 5 minutes).

### `update_status` — State Transition

Transition the agent's status. Agents should only set terminal statuses:

```bash
ait crew status --crew <crew_id> --agent <agent_name> set --status Completed
ait crew status --crew <crew_id> --agent <agent_name> set --status Error
ait crew status --crew <crew_id> --agent <agent_name> set --status Aborted
```

Valid agent-settable statuses: `Completed`, `Error`, `Aborted`. The runner handles `Waiting → Ready → Running` transitions — agents should not set these.

### `update_progress` — Numeric Progress

Report completion percentage (0-100). Can be combined with a status set:

```bash
ait crew status --crew <crew_id> --agent <agent_name> set --progress 75
```

### `read_input` — Read Input Data

Read the agent's input file directly:

```
.aitask-crews/crew-<crew_id>/<agent_name>_input.md
```

This file may be populated by the runner, by upstream agents (writing to a downstream agent's input file), or may be empty.

### `write_output` — Write Results

Write results to the agent's output file:

```
.aitask-crews/crew-<crew_id>/<agent_name>_output.md
```

The runner aggregates outputs from all agents (in topological order) via `ait crew report output`.

### `check_commands` — Poll for Pending Commands

Check if the runner or user has sent commands (kill, pause, resume):

```bash
ait crew command list --crew <crew_id> --agent <agent_name>
```

**Output:** Either `NO_COMMANDS` or the YAML content of pending commands. After reading and acting on commands, acknowledge them:

```bash
ait crew command ack --crew <crew_id> --agent <agent_name>
```

**If a `kill` command is received:** Execute `run_abort_procedure` immediately.

### `run_abort_procedure` — Clean Shutdown

When a kill command is detected or an unrecoverable error occurs:

1. Write any partial results to `_output.md`
2. Set status to `Aborted` (for kill) or `Error` (for errors):
   ```bash
   ait crew status --crew <crew_id> --agent <agent_name> set --status Aborted
   ```
3. Exit cleanly

### `report_error` — Report Error with Message

When an error occurs that the agent cannot recover from:

1. Write the error details to `_status.yaml` via a field-level update (or include in output)
2. Set status to Error:
   ```bash
   ait crew status --crew <crew_id> --agent <agent_name> set --status Error
   ```

Note: The `error_message` field in `_status.yaml` is currently set by the runner (for heartbeat timeouts). Agents reporting errors should include error details in their `_output.md` file.

## Checkpoint Pattern

Checkpoints are points in the work2do where the agent pauses its main work to perform lifecycle operations. They serve three purposes:

1. **Liveness:** Runner detects stuck agents via heartbeat timeout
2. **Control:** User/runner can pause or kill agents between checkpoints
3. **Observability:** Progress tracking and status reporting

### Checkpoint Frequency Guidelines

- **Minimum:** Every 2-3 minutes of wall-clock time
- **After major steps:** Completing a phase, finishing a file, etc.
- **In loops:** Every N iterations (choose N so checkpoints happen every few minutes)
- **Before expensive operations:** Long-running builds, API calls, large file processing
- **After expensive operations:** Confirm success before moving on

### Checkpoint Template

At each checkpoint, execute these operations in order:

```
1. report_alive     — Update heartbeat with progress message
2. update_progress  — Report numeric completion percentage
3. check_commands   — Poll for kill/pause commands
4. [if kill received] → run_abort_procedure and exit
```

## Template work2do Structure

Below is a template showing how to structure a work2do file with checkpoints.

```markdown
# Task: <Agent Name>

## Phase 1: Setup and Validation

Read input from _input.md. Validate prerequisites.
Set up working environment.

### Checkpoint 1
- report_alive: "Phase 1 complete — setup validated"
- update_progress: 10
- check_commands

## Phase 2: Main Work

<Describe the core work to be done>

### Checkpoint 2 (mid-phase)
- report_alive: "Phase 2 in progress — processed N items"
- update_progress: 40
- check_commands

<Continue main work>

### Checkpoint 3 (end of phase)
- report_alive: "Phase 2 complete"
- update_progress: 70
- check_commands

## Phase 3: Output and Finalization

Write results to _output.md.
Validate output completeness.

### Checkpoint 4
- report_alive: "Phase 3 complete — output written"
- update_progress: 95
- check_commands

## Completion

- update_status: Completed
- update_progress: 100
- report_alive: "All phases complete"
```

## Instructions.md: The Mapping Layer

When `ait crew addwork` registers an agent, it generates `<agent>_instructions.md` containing the concrete CLI commands for each lifecycle operation. This file tells the agent exactly which commands to call.

The instructions file is auto-populated with commands specific to the crew ID and agent name, so the agent doesn't need to know its own identifiers — it just follows the instructions.

**Example generated instructions content:**

```markdown
## Status Updates
ait crew status --crew sprint1 --agent coder set --status <status>

## Progress Reporting
ait crew status --crew sprint1 --agent coder set --progress <N>

## Heartbeat
ait crew status --crew sprint1 --agent coder heartbeat --message "..."

## Check Commands
ait crew command list --crew sprint1 --agent coder

## Acknowledge Commands
ait crew command ack --crew sprint1 --agent coder

## Output
Write results to: coder_output.md
```

## Crew-Agnostic Design

The work2do file should reference lifecycle operations by their abstract names only. It should never contain hardcoded crew IDs, agent names, or script paths. This enables:

- **Reuse:** Same work2do template across different crews
- **Portability:** Work2do files can be version-controlled independently
- **Flexibility:** Different agent types can use different CLI wrappers

The `_instructions.md` file is the bridge — it is crew-specific and agent-specific, mapping abstract names to concrete commands.
